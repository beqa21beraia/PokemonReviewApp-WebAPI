# 🎮 PokemonReviewApp

A RESTful Web API built with **ASP.NET Core** that allows users to manage and review Pokémon. This project was built as a learning exercise to practice backend development concepts including repository pattern, async programming, structured logging, and more.

## Overview

PokemonReviewApp is a clean, well-structured REST API where you can:

- 📦 Manage **Pokémon**, **Categories**, **Countries**, **Owners**, **Reviewers**, and **Reviews**
- ⭐ Get **ratings** for Pokémon based on their reviews
- 🔗 Query related data (e.g. Pokémon by category, owners by country)
- 📝 Full **CRUD** operations on all resources

## 🚀 Getting Started

### Prerequisites

- [.NET 8 SDK](https://dotnet.microsoft.com/download)
- [SQL Server](https://www.microsoft.com/en-us/sql-server/sql-server-downloads)
- [Visual Studio 2022](https://visualstudio.microsoft.com/) or [VS Code](https://code.visualstudio.com/)

### Installation

1. **Clone the repository**
   ```bash
   git clone https://github.com/beqa21beraia/PokemonReviewApp.git
   cd PokemonReviewApp
   ```

2. **Configure the connection string** in `appsettings.json`
   ```json
   "ConnectionStrings": {
     "DefaultConnection": "Server=YOUR_SERVER;Database=PokemonReviewDb;Trusted_Connection=True;"
   }
   ```

3. **Apply migrations**
   ```bash
   dotnet ef database update
   ```

4. **Run the application**
   ```bash
   dotnet run
   ```

5. **Open Swagger UI**
   ```
   https://localhost:{port}/swagger
   ```

## 🗄️ Database Setup

The app uses **Entity Framework Core** with **SQL Server**. The database schema is managed via EF Core migrations.

To seed the database with initial data, the project includes a `Seed.cs` class that runs on startup if the database is empty.

To create a new migration after model changes:
```bash
dotnet ef migrations add MigrationName
dotnet ef database update
```

## 📊 Logging

This project uses **Serilog** for structured logging with two sinks:

- 🖥️ **Console** — colored output while running locally
- 📁 **File** — stored in `/Logs/log-YYYYMMDD.txt`, rolling daily

HTTP requests are automatically logged via `UseSerilogRequestLogging()`. Additional manual logs are placed in controllers for key business events:

| Log Level | When used |
|---|---|
| `Information` | Successful create / delete operations |
| `Warning` | Resource not found (404), duplicate resource (422) |
| `Error` | Unexpected server-side failures (500) |

Log files are generated automatically in the `/Logs` folder at the project root.
